# The Journey: Cross-Language Performance Analysis with Claude

## How This Started

This project began as a simple experiment: could Claude (Anthropic's AI) solve
Project Euler problems across multiple programming languages, and how would the
solutions compare? What started as **claude-vs-euler** — a single repository
with Go, Rust, and C solutions — evolved into a systematic benchmark suite
spanning nine languages and over 900 individual solution files.

Every solution in this project was generated by **Claude Opus 4.6**. The human's
role was architectural: choosing languages, defining the benchmark harness
contracts, deciding when algorithms needed normalization, and asking the right
questions about what the data means.

## The Languages

We started with three compiled languages and expanded outward:

| Language | Why It's Here |
|----------|--------------|
| **C** | The baseline. Closest to the metal without writing assembly. Minimal runtime overhead. |
| **C++** | Does the STL abstraction layer cost anything vs raw C? |
| **Rust** | Modern systems language. Shares LLVM backend with Clang — is the performance difference the language or the optimizer? |
| **Go** | Garbage-collected but compiled. How much does GC cost for computational work? |
| **Java** | JIT-compiled on the JVM. Does warmup help it catch compiled languages? |
| **C#** | .NET's RyuJIT. Similar niche to Java — how do the runtimes compare? |
| **Python** | The interpreted baseline. How far behind is developer productivity's favorite language? |
| **Haskell** | Purely functional, lazy evaluation. Does the paradigm impose a performance tax? |
| **ARM64 Assembly** | The floor. Zero abstraction, zero runtime. What's theoretically possible on this hardware? |

### The One We Dropped

**APL** was originally in the mix as an exotic, array-oriented language. We
dropped it for practical reasons: Dyalog APL (the main implementation) wasn't
available on the test machine, and GNU APL uses different enough syntax that the
solutions weren't portable. More fundamentally, APL's strength is in concise
array operations, not the kind of loop-heavy algorithmic work that dominates
Project Euler. It was adding complexity without adding insight.

## Lessons Learned Along the Way

### Algorithms vs. Languages

The single most important realization: **algorithm differences dwarf language
differences**. Early benchmark runs showed Problem 095 (Amicable Chains) taking
7 seconds in C while everything else was under 350ms. The culprit wasn't the
language — it was the benchmark harness calling `solve()` repeatedly, and the
solution rebuilding a million-element sieve on every call.

After adding a one-time initialization guard, the same problem dropped to 78ms.
A **90x speedup** from a one-line fix, in C — the "fast" language.

This pattern repeated across the project. Haskell's Problem 095 used per-number
trial division (`O(sqrt(N))` per number) instead of a sieve (`O(N log N)`
total). That's not a language performance issue — it's a fundamentally different
algorithm. We had to normalize algorithms across all nine languages before any
cross-language comparison was meaningful.

**Rule**: Before comparing languages, make sure you're comparing the same
algorithm. Otherwise you're benchmarking the programmer (or the AI), not the
language.

### The LLVM Confound

C (via Clang) and Rust both use LLVM as their backend optimizer. This means
similar performance between C and Rust might tell us more about LLVM than about
either language. To properly isolate language overhead, we'd need to test C
with GCC as well — same language, different optimizer. That's a future
direction.

### Benchmark Harness Design

Getting consistent, comparable benchmarks across nine languages with different
compilation models required careful design:

- **Compiled languages** (C, C++, Rust, Go): compile once, run many
- **JIT languages** (Java, C#): need warmup iterations before timing
- **Interpreted** (Python): no warmup helps, but consistency matters
- **Assembly**: linked with a C timing harness

Every harness follows the same contract:
1. Warmup runs (3 iterations)
2. Calibration run to determine iteration count
3. Timed runs with median reporting
4. Output: `BENCHMARK|problem=NNN|answer=ANSWER|time_ns=TIME|iterations=ITERS`

The benchmark scripts parse this output into JSON for aggregation.

### Data File Handling

Several Project Euler problems require input data files (matrices, word lists,
number grids). Each language handles this differently:

- **C/C++**: `fopen`/`fread` with static initialization guards
- **Rust**: `include_str!` or `OnceLock` with file reads
- **Go**: `//go:embed` directive
- **Java/C#**: Lazy-cached static fields
- **Python**: Module-level cached variables
- **Haskell**: `IO` monad with `unsafePerformIO` or pre-loaded in `main`
- **ARM64**: Data embedded directly in `.data` section

The key insight: file I/O must happen **outside** the timed region, or you're
benchmarking disk speed, not computation.

## Platform Notes

All benchmarks run on **Apple Silicon** (M-series, ARM64). This matters:

- ARM has different optimization profiles than x86
- Some compilers are more mature on one architecture
- Cloud workloads (AWS, GCP) may run on x86 Graviton or Intel — results won't
  directly transfer
- Apple's Clang may have Apple-specific optimizations not present in upstream
  LLVM

Future work could include cross-platform comparisons, but that adds a
significant infrastructure burden.

## What's Next

### Compiler Comparison

Once we identify the top-performing languages, the next dimension is **compiler
choice within a language**:

- C/C++: Apple Clang vs GCC vs upstream Clang
- Java: OpenJDK vs GraalVM (especially GraalVM native-image)
- C#: .NET RyuJIT vs NativeAOT vs Mono

### Deeper Metrics

Current benchmarks measure wall-clock time. Future iterations could add:

- **Memory usage** (peak RSS)
- **Source code size** (lines, characters — a proxy for expressiveness)
- **Compilation time** (matters for developer experience)
- **Binary size** (matters for deployment)

### The AI Angle

Every solution was generated by Claude Opus 4.6. An interesting meta-question:
does Claude produce equally idiomatic code across all languages? Are there
languages where the AI-generated code is systematically suboptimal because the
training data had fewer examples of high-performance patterns?

This project doesn't answer that question yet, but the infrastructure is in
place to investigate it.
